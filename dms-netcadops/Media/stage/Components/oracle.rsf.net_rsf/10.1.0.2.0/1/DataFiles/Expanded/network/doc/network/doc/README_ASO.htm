1 INTRODUCTION

This README file is relevant only to the delivered Oracle Advanced Security Release 1 (10.1) - BETA 2. This README documents any differences between Oracle Advanced Security and its documented functionality, as well as known problems and workarounds. Operating system releases, such as UNIX, Windows NT, and so on, often provide an Oracle operating system-specific README document.

For information about upgrading, downgrading, and migration, see Oracle10i Database Upgrade Guide.

Oracle Advanced Security Release 1 (10.1) - BETA 2 bundles security services for the Oracle Database. It secures connections over all protocols into an Oracle Database and integrates a Public Key Infrastructure (PKI). Oracle Advanced Security provides data encryption and integrity for all network protocols into the database, including Oracle Net with native encryption, Oracle Net/SSL, IIOP/SSL, and Java-based encryption for Thin JDBC clients. It integrates with third-party authentication, authorization and single sign-on services. It supports public-key solutions including Secure Sockets Layer (SSL) and X.509, Version 3 certificates (provided by third party certificate servers) and bundles related tools including Oracle Wallet Manager 10.0.0. It also enables SSL-based single sign-on and certificate-based server-to-server authentication over database links.

Oracle Advanced Security utilizes LDAP v3-compliant Oracle Internet Directory for centralized user management. Oracle Enterprise Security Manager 4.0 tools are used to manage enterprise users and enterprise roles. Starting with this release, Oracle Internet Directory is delivered as part of the Oracle Identity Management infrastructure. This release of Oracle Advanced Security is certified to work with Oracle Identity Management infrastructure delivered with the Oracle Database. Directory Integration and Provisioning Platform is also a part of the infrastructure component and allows integration with other enterprise directories including Microsoft Active Directory, SunONE and iPlanet Directory Services.

For more information on the third party directory support, please review the Directory Integration Platform chapter in Oracle Internet Directory Administrator's Guide, which is available on the identity management infrastructure product CD-ROM.

In this release, Oracle Advanced Security installs automatically during Typical Install of the Enterprise Edition of the Oracle database. Oracle Advanced Security can also be installed manually using Custom Install. All components install in Typical Install except DCE, which installs in Custom Install provided the DCE software is already installed on your system.

Although installed by default, Oracle Advanced Security is an extra cost option to the Oracle Database Enterprise Edition and must be purchased when used. This licensing requirement also affects customers wishing to use security features in combination with Java Beans (EJB over IIOP/SSL) or database Enterprise Users (Oracle Net/SSL). The exclusive exception is an HTTPS (HTTP/SSL) connection to the RDBMS which does not require an Oracle Advanced Security license.

Please see the Oracle Advanced Security Administrator's Guide for this release for further information.

2 NEW IN THIS RELEASE

New in this release:

  • Kerberos-Enabled Enterprise Users

    Kerberos-based authentication to the database is available for the first time in this release for centrally managed directory users (in Oracle Internet Directory).

  • Database to Oracle Internet Directory Communication Using Passwords (SSL is no longer mandatory)

    In this release, the database can bind to Oracle Internet Directory by using password SASL-based authentication, rather than requiring PKI certificates for SSL-based authentication. This new functionality removes the overhead of setting up PKI credentials for Oracle Internet Directory and multiple databases.

  • Password Synchronization between Oracle9iAS Single Sign-On and Database Authentication Password

    Starting in this release of Oracle Advanced Security, users' database passwords (orclpassword) are synchronized with the users' Oracle9iAS Single Sign-On passwords. Thus, there is no requirement to set the orclpassword with Enterprise User Security tools, such as Enterprise Security Manager or Enterprise Security Manager Console.

  • Integration with Active Directory by Using the Directory Integration and Provisioning Services

    The Oracle Identity Management Infrastructure includes Directory Integration and Provisioning Services as a means of providing integration with third-party directories, including Microsoft Active Directory and Sun One (iPlanet).

  • Hardware Token Support Using PKCS#11 for SSL

  • Oracle Wallet Manager Integration with Hardware Tokens

  • CRL (Certificate Revocation Lists) and CRLDP (CRL Distribution Point) support for certificate validation

3 INSTALLATION

If using Oracle Advanced Security on a "client only" machine (that is, with no database present), it is mandatory to set the TWO_TASK environment variable before starting the installation. The TWO_TASK variable points to an alias representing the database on a server machine. Setting the TWO_TASK environment variable enables Oracle Advanced Security to be installed in "Client Only" mode. See the Oracle10i Database Administrator's Guide for more details on the TWO_TASK environment variable.

5 EXTERNAL AUTHENTICATION AND SINGLE SIGN-ON

In order to require external authentication and disable username/password authentication, set the sqlnet.ora parameter SQLNET.AUTHENTICATION_REQUIRED=TRUE. The default is false.

The Cybersafe adapter is desupported starting with the Oracle Advanced Security Release 1 (10.1). Oracle recommends using Oracle's Kerberos adapter in its place. Kerberos authentication with the Cybersafe KDC (Trust Broker) continues to be supported when using the Kerberos adapter.

5.1 Changes to the Startup Command

In this release, if external authentication (using Kerberos, RADIUS) is not enabled, please verify that you have either issued the startup command with a PFILE option so that the parameters from your init<SID>.ora are picked up.

6 RADIUS ADAPTER

Oracle Advanced Security supports RADIUS-compliant servers and authentication devices. As in prior releases (Oracle Advanced Security 9.1 and 9.2), RSA ACE/Server and tokens can authenticate Oracle users only through the RADIUS adapter. To use the Java-based client interface for RADIUS, you must include $ORACLE_HOME/JRE/lib/sparc/native_threads in the LD_LIBRARY_PATH environment variable. To use RADIUS, you must use native threads. Set the variable THREADS_FLAG to "native" within the Java Runtime Environment.

The following bugs and workarounds apply to the RADIUS adapter in this release:

  • The CHAP (challenge-response) mode is disabled for Beta 2, so the challenge-response default interface does not function. [Reference Bugs 2793023, 2904046]

  • On the Windows platform, during installation, if you configured the RADIUS adapter, please reboot the machine to get the JRE location. [Reference Bug 2212844]

7 SECURE SOCKETS LAYER

For the SSL adapter, dynamic specification of sqlnet.ora parameters such as SSL_VERSION, SSL_CIPHER_SUITES, and SSL_CLIENT_AUTHENTICATION, as part of TNS Aliases, will not have any effect on the redirected connections made to the database server.

There is a known bug in which an OCI client requires a wallet even when using a cipher suite with DH_anon which does not authenticate the client. Such cipher suites are known to be vulnerable to person-in-the-middle attacks. If you use a cipher suite with DH_anon, you should use Oracle Advanced Security native encryption and checksumming to protect against the attack. [Reference Bug 1661031]

DH_anon_export cipher suite for SSL is not supported in this release (Release 1 (10.1) - BETA 2).

8 ORACLE WALLET MANAGER 3.0

Oracle Wallet Manager Online Help becomes unresponsive when modal dialog boxes, such as the one to enter certificate request information, pop up. The Online Help becomes responsive once the modal dialog box is closed. [Reference Bug 1114710]

9 JAVASSL AND JSSE

Oracle Corporation recommends using the jsse.jar provided by your platform vendors for Java SSL requests. Oracle JavaSSL is desupported starting in this release 10.0.0.

10 ENTRUST SUPPORT

This release supports Entrust version 5.0.2, 5.1 and 6.0 components, including IPSEC Negotiator Toolkit, Entrust/Authority and Server Login. On HP-UX 64-bit, Solaris 64-bit, and Compaq Tru 64 platforms, Oracle Advanced Security supports Entrust 6.0 PKI.

On Windows, you must install Entrust Entelligence on the client.

You must have the same version of Entrust tool kits and the Entrust Authority if you are using 5.x versions of Entrust. For example, you can use 5.1 version everywhere or 5.0.2 version for the tool kits and the EA. However, you can have Entrust 6.0 IPSEC Negotiator and Server Login tool kit with 5.1 Entrust Authority.

In this release (just as in 9.0.1), you do not have to choose between Entrust or SSL upon installation. In order to enable Entrust-based connections, follow the instructions in the Oracle Advanced Security Administrator's Guide.

11.1.1 Enterprise Security Manager Changes, Bugs, and Workarounds

The following changes, bugs, and workarounds apply to the Release 1 (10.1) - BETA 2 version of Enterprise Security Manager:

  • In this release, some of the Enterprise Security Manager functionality is integrated into the Enterprise Security Manager Console, a browser-based graphical user interface tool based on the Oracle Delegated Administration Service (DAS) Framework. The Enterprise Security Manager Console is launched from the Operations menu of the Enterprise Security Manager main application window. See: "New Tool: Enterprise Security Manager Console"

  • In previous releases, Enterprise Security Manager allowed login by using nickname/userid or distinguished name (DN), but in this release only login by DN is fully supported. Login by nickname will work in some circumstances only. [Reference Bug 2810000]

  • Do not delete the OracleDefaultDomain. Enterprise Security Manager allows you to delete the OracleDefaultDomain. However, this special domain is required for database registration, so do not delete it. [Reference Bug 1995553]

  • Logging in as cn=orcladmin fails if anonymous binds are disallowed in Oracle Internet Directory. [Reference Bug 3030547]

11.1.2 New Tool: Enterprise Security Manager Console

The Enterprise Security Manager Console, which is based on the Oracle Internet Directory Delegated Administration Services (DAS), is new in this release. Administrators can use this tool to create enterprise users, enterprise user security groups, and to configure identity management realm attributes that relate to enterprise user security. See: Chapter 13 in the Oracle Advanced Security Administrator's Guide for information about using this new tool.

11.1.3 Enterprise Login Assistant Replaced

In this release, Enterprise Login Assistant functionality has been migrated to the new Enterprise Security Console and Oracle Wallet Manager. The following table lists which tool you should now use to perform tasks that were formerly performed by using Enterprise Login Assistant:

If you used Enterprise Login Assistant to... Then now you should use...

change the directory-database password

Enterprise Security Manager Console

change a wallet password

Oracle Wallet Manager

enable auto login for a wallet

Oracle Wallet Manager

These tasks are documented in the Oracle Advanced Security Administrator's Guide.

11.1.4 Oracle Net Configuration Assistant No Longer Used

This tool is no longer used to set up Oracle home directory access or to create an ldap.ora file. The options still appear in this tool's user interface, but they do not function properly. In the future, Oracle Internet Directory Configuration Assistant will be used for this purpose, but for this release (Release 1 (10.1) - BETA 2), you must manually create an ldap.ora file in the $ORACLE_HOME/network/admin directory. See: "Manual Creation of ldap.ora Required for Directory Access" for information about creating an ldap.ora file. Note: You must create the ldap.ora file before you register a database in the directory with Database Configuration Assistant. [Reference Bugs 2826399, 2809510]

11.2 Directory Changes and Issues that Affect Enterprise User Security

The following directory changes and issues affect Enterprise User Security in this release:

  • Default Nickname Attribute Change

  • orcladmin User Identity Changes

  • User Security Admins Group Limited Functionality

  • Adding Oracle Internet Directory Groups to Enterprise Roles Only Supported for GroupOfUniqueNames Objectclasses

11.2.1 Default Nickname Attribute Change

The default nickname attribute for an Oracle Context in 9.2 Oracle Internet Directory used to be cn, was set in the root Oracle Context only, and applied to all Oracle Contexts in the directory. In this release, the 9.0.4 Oracle Internet Directory, the default value for this attribute is uid, it is set in each Identity Management Realm, and applies to that realm only.

11.2.2 orcladmin User Identity Changes

In previous releases, there was only one orcladmin user (cn=orcladmin) who was a "virtual" identity possessing root privileges in Oracle Internet Directory. It was a "virtual" identity because there was no directory entry representing this user. In the current release of Oracle Internet Directory (release 9.0.4), the "virtual" orcladmin user who possesses root privileges throughout the directory still exists, but each release 9.0.4 Identity Management Realm (called "Subscriber" in previous releases) now also includes, by default, an orcladmin user who is intended to be the root user for that realm only. These realm-specific orcladmin users do have directory entries that represent them: cn=orcladmin,cn=Users,<realm_DN>.

If you use orcladmin as your username when you use Oracle9iAS Single Sign-On to log in to a tool that is based on Oracle Internet Directory Delegated Administration Services (DAS), then the Single Sign-On server will log you in as the realm-specific orcladmin for the default identity management realm (and not as the all-powerful "virtual" root user). Consequently, when you are logged in as the realm-specific orcladmin user, you can use a DAS-based tool to manage data only within that realm, and not in the rest of the directory. This may cause confusion if you are accustomed to orcladmin having full access to all data in Oracle Internet Directory. Note: Tools that are based on Oracle Internet Directory Delegated Administration Services include Enterprise Security Manager Console and Oracle Internet Directory Self-Service Console [Reference Bug 3057246]

11.2.2.1 Managing Other Identity Management Realms

If you are in the appropriate tool administrative group or you are logged in as orcladmin, then you can use a DAS-based tool to manage the directory data in your identity management realm, which is determined during the Oracle9iAS Single Sign-On login. If you have the appropriate Access Control List (ACL) permission in Oracle Internet Directory, then you can also view other identity management realms. To manage other identity management realms by using a DAS-based tool, then you must log out with Oracle9iAS Single Sign-On and log in as an administrative user in the other realm.

[Reference Bug 3027410]

11.2.3 User Security Admins Group Limited Functionality

This group is not fully functional in this release. [Reference Bugs 2942611, 2942598]

11.2.4 Adding Oracle Internet Directory Groups to Enterprise Roles Only Supported for GroupOfUniqueNames Objectclasses

Adding directory groups to enterprise roles is a way to grant those roles to groups of users. To enable this functionality, the directory groups must have the GroupOfUniqueNames objectclass with members in the uniquemember attribute. Note: the GroupOfNames objectclass and its corresponding member attribute are not supported in this, nor in previous releases. [Reference Bug 3038229]

11.3 Manual Creation of ldap.ora Required for Directory Access

No ldap.ora is created by Oracle Internet Directory Configuration Assistant or Oracle Net Configuration Assistant, and there is no sample ldap.ora available.

Workaround: Create an ldap.ora manually using the sample below:

# sample ldap.ora file:

DIRECTORY_SERVERS= (<directory_host>:<non-SSL_port>:<SSL_port>)

DEFAULT_ADMIN_CONTEXT = "realm_DN"

DIRECTORY_SERVER_TYPE = OID

[Reference Bugs 2819512, 2809510, 2826399, 2923046]

Note: The ldap.ora file must be created before you register a database in the directory.

11.5 Enter Wallet Location into Database sqlnet.ora File for Database to Directory Password Authentication

In this release the database can bind to Oracle Internet Directory by using password SASL-based authentication, making it unnecessary to have PKI certificates for SSL-based authentication. Although PKI certificates are not required, the database still must have a wallet to hold its credentials for communicating with the directory. In this release, Database Configuration Assistant automatically creates an appropriate database wallet in $ORACLE_HOME/admin/<SID>/wallet during database registration, and configures it with proper credentials.

Because the wallet is required, the database sqlnet.ora file must contain a wallet location.

To use Oracle Net Manager to set the wallet location in your database's sqlnet.ora file:
  1. In the Oracle Net Manager navigator pane, select Profile. A drop-down list appears in the right pane.

  2. In the right pane, choose Oracle Advanced Security from the list. A set of property sheets displays in the right pane.

  3. Choose the SSL tab. The SSL property sheet displays.

  4. In the SSL property sheet, check Configure SSL for Server

  5. In the Wallet Directory field, change the default directory location to $ORACLE_HOME/admin/<SID>/wallet

  6. Save your network configuration changes.

11.6 Enable SSL in No Authentication Mode in Oracle Internet Directory for Password Authentication between the Database and the Directory

For password-based authentication between the database and Oracle Internet Directory, enable SSL with no authentication in the directory by using Directory Manager. Refer to Oracle Internet Directory Administrator's Guide, which is available on the identity management infrastructure product CD-ROM, for information about how to use Directory Manager to configure this.

11.7 Error ORA-28301 Produced When Database Is Not Registered in an Enterprise Domain

If you receive the error message ORA-28301: Domain Policy hasn't been registered for SSL authentication during an enterprise user login to the database using SSL (client to database connection uses SSL), then ensure that the database is registered in an enterprise domain in the directory. The database must be registered in an enterprise domain for an Enterprise User Security login to work. [Reference Bug 3076879]

11.8 Database Must Be Restarted When Enterprise Domain Membership Changes

In some cases, when a database is moved from one enterprise domain in the directory to another, the database must be restarted for the change to be reflected in an Enterprise User Security login. [Reference Bug 3076903]
